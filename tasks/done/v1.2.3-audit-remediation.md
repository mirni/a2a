### P0 — v1.2.3 (ship within 24 hours)

#### T-1 **CRIT-1: Wire verifier client in production deployment**

**Root cause:** `gateway/src/lifespan.py:202-219` only instantiates `VerifierClient` when `VERIFIER_AUTH_MODE` is set. Production sandbox has this env var missing, so `verifier_client=None`, `GatekeeperAPI` has no backend, and every job fails synchronously.

**Fix (infra, no code change):**
1. Add the following to the a2a-gateway systemd unit env file (`/etc/a2a/gateway.env` or `EnvironmentFile=` path):
   ```
   VERIFIER_AUTH_MODE=iam
   VERIFIER_LAMBDA_FUNCTION=z3-verifier
   VERIFIER_LAMBDA_REGION=us-east-1
   ```
2. Restart service. Confirm with:
   ```bash
   curl -sS -X POST https://api.greenhelix.net/v1/gatekeeper/jobs \
     -H "Authorization: Bearer <PRO>" \
     -H 'Content-Type: application/json' \
     -d '{"agent_id":"<pro-agent>","properties":[{"name":"p","language":"z3_smt2","expression":"(declare-const x Int) (assert (> x 0)) (check-sat)","scope":"economic"}],"scope":"economic","timeout_seconds":30}'
   ```
   Expect `"status":"completed"`, `"result":"satisfied"`.
3. Also fall back to `VERIFIER_AUTH_MODE=mock` as a safety default if IAM credentials aren't provisioned — log a WARN but keep the service functional against in-process Z3.

**Code change (defensive default):** in `gateway/src/lifespan.py`, if `VERIFIER_AUTH_MODE` is empty, default to `"mock"` in sandbox and log an ERROR in prod. This way we can never ship again with `verifier_client=None`.

**Acceptance:**
- [ ] Sandbox `POST /v1/gatekeeper/jobs` with the OpenAPI example returns `status:"completed"` within the timeout.
- [ ] Job list shows `result` field set to `"satisfied"` / `"violated"` / `"unknown"` — not `null`.
- [ ] Healthcheck at `/v1/health` includes `gatekeeper: "ok"`.
- [ ] Regression test `gateway/tests/v1/test_gatekeeper_live.py` exercises end-to-end submit→completed against `MockVerifierClient`.

**Size:** S (env var + 1 config line + 1 defensive default + 1 integration test).

#### T-2 **CRIT-2: Zero-out `cost` field in gatekeeper response when job fails**

**Root cause:** `gatekeeper.py:124` waives the wallet debit via `tc.cost = 0.0` but `finalize_response` still echoes `result.cost` from the job record, so the response says `cost:"6"` even though the wallet wasn't debited. Audit personas read this as "you charged me."

**Fix:**
- In `gateway/src/routes/v1/gatekeeper.py` submit handler, after the `tc.cost = 0.0` line, also mutate the response dict: `result["cost"] = "0"` (keep as string to match existing response schema).
- Add a second field `result["billed_cost"] = "0"` for absolute clarity.
- Update the OpenAPI spec for `VerificationJob` to document that `cost` reflects the actual amount debited, not the catalog list price.

**Acceptance:**
- [ ] Failing job returns `cost:"0" billed_cost:"0" status:"failed"`.
- [ ] Successful job returns `cost:"6" billed_cost:"6" status:"completed"`.
- [ ] Regression test in `gateway/tests/v1/test_audit_v1_2_2_regressions.py::TestGatekeeperCostZeroOnFailure` (TDD — RED first).

**Size:** XS (two-line code change + test).

**Acceptance:**

- [ ] Add explicit assertion in `test_audit_v1_2_2_regressions.py::TestInfraKeysNotLeakingFleet`.

### P1 — v1.2.3 (same release, ship within 48 hours)

#### T-6 **HIGH-7: Add grace window + confirmation to key rotation**

**Root cause:** `POST /v1/infra/keys/rotate` immediately invalidates the old key and emits a new one, with no grace period. The endpoint is so trivially reachable that two consecutive auditor agents tripped it accidentally during schema discovery.

**Fix:**
- Add `grace_seconds` field to the rotation storage schema (default 300s).
- During validation, accept the old key for `grace_seconds` after rotation, returning a `X-A2A-Key-Grace-Expires-At` header to warn clients.
- Require a `X-Rotate-Confirmation: confirm` header on the rotate endpoint (belt-and-braces — an accidental `POST /v1/infra/keys/rotate {}` without this header returns 428 Precondition Required).
- Add telemetry: log `key_rotation_event` with the caller's `agent_id` + source IP.
- Whitelist `audit-*` and `test-*` agent_id prefixes from auto-revoke heuristics.

**Acceptance:**
- [ ] Old key continues to validate for 300s after rotation, returning a `Warning` header.
- [ ] Rotation without `X-Rotate-Confirmation` header → 428.
- [ ] Audit-labelled keys are never auto-revoked by revocation heuristics.
- [ ] Regression test `test_key_rotation_grace_window.py`.

**Size:** M (schema migration + middleware + 4 tests).

#### T-7 **HIGH-5: Seed CREDITS↔{BTC,ETH,...} default rates + fix round-trip precision**

**Root cause:** `initialize_default_rates()` seeds ETH→CREDITS but not CREDITS→ETH, and even with the v1.2.2 two-hop pivot, 100 CREDITS → ETH → 100 CREDITS loses everything because the ETH amount rounds to `0.00`.

**Fix:**
- In `products/billing/src/exchange.py::initialize_default_rates`, seed BOTH directions for every supported pair.
- In `ExchangeRateService.convert`, bump the internal working precision to 18 decimals (ETH precision), round to the target currency's `decimal_places` only on final return.
- Regression test: `test_multi_currency.py::TestExchangeRateService::test_round_trip_preserves_value_above_dust_threshold` — 100 CREDITS → ETH → CREDITS ≥ 99.99 CREDITS.

**Acceptance:**
- [ ] `GET /v1/billing/exchange-rates?from_currency=CREDITS&to_currency=ETH` returns non-zero rate.
- [ ] Round-trip 100 CREDITS → ETH → CREDITS preserves ≥ 99 % of value.
- [ ] 18-decimal working precision used internally for all ETH-denominated calcs.

**Size:** S (seed data + precision tweak + 3 tests).

**Root cause:** `POST /v1/infra/keys` creates the API key but doesn't insert a row into the identity service. `/v1/identity/agents/{id}/reputation` 404s until the owner calls `POST /v1/identity/agents` themselves.

- Regression test: create key → immediately call `GET /v1/identity/agents/{id}/reputation` → expect 200.

**Acceptance:**

- [ ] Existing agent key creation is idempotent (no duplicate identity error).

#### T-11 **MED: Idempotency-Key with different body returns 409**

**Root cause:** Gateway currently accepts an `Idempotency-Key` header but only uses it to dedupe inside the same transaction — different bodies under the same key create new resources.

**Fix:**
- Add `idempotency_keys` table: `(agent_id, key, body_hash, response_body, response_status, created_at)`.
- Middleware: on any write request with `Idempotency-Key`, compute `body_hash=sha256(request.body)`, look up existing row.
  - Same body → return cached response
  - Different body → 409 `{"type":".../idempotency-mismatch"}`
- TTL: 24h.

**Acceptance:**
- [ ] `POST /v1/payments/intents` twice with same key + same body → same response, single resource.
- [ ] Same key + different body → 409.
- [ ] Test coverage in `test_idempotency.py`.

**Size:** M (schema migration + middleware + 5 tests).

#### T-14 **MED: Bimodal latency — warm SQLite connection pool at startup**

**Root cause:** p99 is 5.28s on 26 % of `/v1/health` calls. Almost certainly a cold-SQLite-connection issue: first request per idle period re-opens the DB file.

- Add a `slow_request` metric at the middleware level: log any request > 1s.

**Acceptance:**
- [ ] p99 latency on `/v1/health` ≤ 500ms across 100 sequential samples.
- [ ] Metric `slow_requests_total` exposed at `/metrics`.

#### T-16 **MED: Gatekeeper cancel endpoint — make truly async or remove**

**Root cause:** After T-1 is deployed, gatekeeper jobs will be async (Lambda). Until then, they complete synchronously in <250ms, making the `cancel` endpoint unreachable.

#### T-17 **HIGH-NEW: Multi-agent topology support for single PRO key**

**Root cause:** A PRO key is scoped to one `agent_id`. Devs building ML workflows (buyer/seller/judge) can't simulate the flow from one key.

**Fix (architectural — pick ONE):**
- **Option A (recommended):** Add a "development sandbox" concept. A PRO key can create up to 10 "child agents" (`/v1/dev/child-agents`) that it acts on behalf of. Child agents inherit the PRO tier's limits and billing, but have their own `agent_id` for ownership checks.
- **Option B:** Extend `/v1/identity/agents` to allow a PRO key to register new agents (only if `parent_agent_id == tc.agent_id`).
- **Option C (simplest):** Document that multi-agent flows require multiple PRO keys and offer a bulk-provisioning endpoint for enterprise tier.

**Deliverables:**
- [ ] ADR: `docs/adr/NNNN-multi-agent-dev-topology.md`.
- [ ] Implementation of chosen option.
- [ ] Quickstart update in `docs/quickstart/multi-agent.md`.
- [ ] SDK helper: `client.create_dev_child_agent(name)`.

**Size:** L (1 week).

### v1.2.3 PR strategy

- Single PR, branch `fix/audit-v1.2.2-remediation`.
- Each task gets its own commit with a TDD red→green→refactor cycle documented in the commit body.
- Regression tests live in `gateway/tests/v1/test_audit_v1_2_2_regressions.py` (mirrors the v1.2.1 file).
- PR body references each task ID and the concrete audit evidence file (`reports/external/v1.2.2/.../run.log` line numbers).
- CI must be all green before merge (including staging deploy).
- After merge: tag `release/1.2.3` and run Deploy Production workflow.
- **Before deploy**: set `VERIFIER_AUTH_MODE=iam` on production host (T-1). Ship the env file change first via Ansible/systemd or checked-in config, then deploy the gateway binary.
- Post-deploy smoke test: reproduce the audit's `POST /v1/gatekeeper/jobs` call and confirm `status:"completed"`.

### Out of scope for v1.2.3

- T-4 fixture check is informational; it's not a blocking code change.
- T-17 (multi-agent topology) is architectural — targets v1.3.0.
- T-22/T-23/T-24 are infra hygiene — do as a separate `chore/infra-hygiene` branch.
