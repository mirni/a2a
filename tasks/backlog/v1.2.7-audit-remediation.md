# v1.2.7 Audit Remediation

## Context

External audit `reports/external/v1.2.7/multi-persona-audit-v1.2.7-2026-04-11.md`
— **NO-GO, 5.5/10 (up from 5.0)**. This is the **most improved release**
on the security axis in the audit history: **6 of 7 persistent v1.2.3/v1.2.4
CRITs are fixed** (admin gate, IP-spoof headers, X-Api-Key, header
normalization, path normalization). Security persona score jumped 3 → 9.

However, a **new CRITICAL** was found: budget enforcement reads
`cap=0.00` regardless of configured value and blocks ALL paid endpoints
with 402 even when wallets are funded and budgets are set high. This
single bug masks whether other fixes actually work and must be rooted
out before the next audit cycle.

The sections below organise every remaining finding by severity. Each
item links back to the persona that raised it in the audit report.

---

## P0 — blocks next release

### P0-1 · Budget enforcement reads `cap=0.00` (NEW CRITICAL)

**Report ref:** `NEW-CRIT-7-2-7`, raised by ML, Web3, SRE personas.

> Budget enforcement reads `cap=0.00` regardless of configured value.
> Blocks ALL paid endpoints (gatekeeper, intent create, escrow create,
> marketplace match) with 402 even when wallet has funds. `PUT` writes
> to one store, runtime reads from another.

**Hypothesis:** `PUT /v1/billing/wallets/{id}/budget` persists to one
store (paywall DB? wallet metadata?) while the capture-path enforcer
reads from another (agent metadata cache? config table?). The write
and read paths need to be unified onto a single canonical source.

**Acceptance:**
- Red test: set budget_cap=100 via PUT, call paid endpoint costing 1,
  assert 200 (currently 402).
- Red test: set budget_cap=100 via PUT, call paid endpoint costing 101,
  assert 402 with body containing `cap=100.00`, not `cap=0.00`.
- Concurrent write + read race test (already specced in
  `tasks/backlog/v1.2.4-audit-p1.md` P0-5 follow-up).
- Sandbox-parity: add `tests/sandbox/test_budget_cap_round_trip.py`
  that PUTs a cap and immediately hits a paid endpoint.

### P0-2 · Gatekeeper Z3 engine 100% failure (5th release)

**Report ref:** `CRIT-Gatekeeper`, raised by ML + Web3.

> 5th consecutive release with Gatekeeper completely broken (0/50
> successful jobs). 84/84 historical jobs `result=error`. Pull from
> pricing or fix the solver.

**Hypothesis:** Either the Z3 engine bootstrap is broken in the
packaged image (missing system library, Python ABI mismatch), or the
verifier Lambda is returning errors we silently forward. Needs
root-cause triage — install `z3-solver` was fixed in the CI workflow
(34ada4d) but the runtime image may still be missing it.

**Acceptance:**
- Reproduce in a clean container pull: `docker run greenhelix/a2a-gateway:1.2.7`
  then POST a minimal gatekeeper job and observe the failure.
- Fix runtime dep (Dockerfile) OR route to the real Lambda.
- Red test: contract test that a trivial job (`sum(x) >= 0`) returns
  `result=ok` end-to-end.
- If the solver cannot be fixed this cycle, **pull Gatekeeper from
  pricing** and flag it NOT-GA. The audit has asked for this twice.

### P0-3 · `/v1/identity/agents` POST restricted to enterprise

**Report ref:** `HIGH`, raised by Fintech + ML.

> `/v1/identity/agents` POST restricted to enterprise tier — free/pro
> both 403. Breaks self-serve agent onboarding for non-enterprise
> customers.

This is a regression from prior releases where self-serve agent
creation was allowed. The tier gate in the catalog or
`ADMIN_ONLY_TOOLS` / `require_tool` chain is too strict.

**Acceptance:**
- Red test: free key POSTs to `/v1/identity/agents` → assert 201, not 403.
- Red test: pro key POSTs to `/v1/identity/agents` → assert 201, not 403.
- Sandbox-parity regression: extend `test_audit_probe.py`.
- Verify PRO key can still add a second agent (multi-agent topology).

### P0-4 · SDK response field naming mismatch (`id` vs `intent_id`)

**Report ref:** `HIGH`, raised by Fintech.

> SDK 1.2.7 `PaymentIntentResponse` exposes `id` (not `intent_id`);
> `SubscriptionResponse` exposes `id` (not `subscription_id`). Breaks
> any caller using attribute access on `.intent_id` / `.subscription_id`.

**Decision required:** do we keep `id` (modern, uniform) or restore
`intent_id`/`subscription_id` (historical, breaks if dropped)? The
safe path is to keep `id` as the canonical field **and** add
`intent_id`/`subscription_id` as aliases via
`Field(alias=..., validation_alias=AliasChoices(...))` with model-dump
emitting both keys for one more release, then drop in v1.3.0.

**Acceptance:**
- Contract tests covering both attribute paths (`.id` and `.intent_id`).
- Deprecation warning on the legacy alias.
- SDK changelog entry.

### P0-5 · SSE heartbeats still missing (regressed since v1.2.3)

**Report ref:** `MED-6`, persistent across v1.2.4/v1.2.5/v1.2.6/v1.2.7.

> `IncompleteRead(13 bytes)`, no heartbeat. Streams close without the
> periodic `: keep-alive\n\n` frames clients expect.

**Acceptance:**
- Red test: open an SSE stream, assert at least one heartbeat frame
  arrives within 15s.
- Fix the writer to emit `keep-alive` at a configurable interval
  (default 10s).
- Regression test at both the unit and sandbox-parity layer.

### P0-6 · Restore or alias missing admin routes

**Report ref:** raised by SRE persona.

> Several admin routes (`/v1/agents/rotate_key`, `/v1/audit/log`,
> `/v1/infra/databases/integrity_check`) return 404. Possibly renamed
> without v1 aliases.

The three-way infra router split that landed in the v1.2.4 P0 PR may
have dropped these paths. They are present in the OpenAPI spec but
missing at runtime.

**Acceptance:**
- Enumerate all missing routes via `scripts/dump_openapi.py` →
  diff against what actually responds.
- Restore the handlers or publish `310 Gone` / alias to the new path.
- Route-enumeration contract test (T-2 from
  `tasks/backlog/v1.2.4-audit-p1.md`) would have caught this — wire
  it up this cycle.

## P1 — next cycle

### P1-1 · Refund bookkeeping $1 balance drift

**Report ref:** `HIGH`, raised by Fintech.

> Full refund produced +$1 balance drift in one test (`b1=1001` vs
> `b0=1000`); race condition or double-credit.

Needs a replay test with `asyncio.gather` to catch the race. Suspect
the fee-retention path or a double-credit on partial→full refund
upgrade.

### P1-2 · USDC/USDT/DAI/SOL/JPY exchange-rate pairs 500

**Report ref:** `MED-USDC`, raised by Web3.

> USDC exchange-rate 500 — scope widened in v1.2.7 to USDT, DAI, SOL,
> JPY. Previously only USDC was broken.

Likely a regression in the FX provider integration. Root-cause and
add a contract test that touches every stablecoin pair in the
supported set.

### P1-3 · Prometheus `/metrics` + `/livez`/`/readyz` probes

**Report ref:** raised by SRE persona, **5 consecutive cycles unfixed**.

> No Prometheus `/metrics`, no `/livez`/`/readyz`. Can't run on k8s.

Add three endpoints:
- `GET /metrics` — Prometheus text format (already partially implemented
  at `/v1/metrics` but returns plain JSON 403 — needs RFC 9457 and
  the standard path).
- `GET /livez` — always 200 if the process is up.
- `GET /readyz` — 200 only if DB + paywall + catalog reachable.

### P1-4 · ML metric allow-list (PRO key 403)

**Report ref:** `HIGH-METRIC-LIST`, raised by ML.

> PRO key now hits 403 InsufficientTier on `/v1/identity/metrics/ingest`
> (allow-list can't even be tested). WORSE than v1.2.4.

The tier check fires before the allow-list check. Relax the tier gate
on `metrics/ingest` or move the allow-list in front of the tier gate.

### P1-5 · Wallet auto-create on agent register

**Report ref:** `HIGH-WALLET-AUTOCREATE`, persistent since v1.2.4.

> Agent register no longer creates wallet; explicit
> `POST /v1/billing/wallets` required.

Restore the onboarding convenience: agent register should atomically
provision a wallet.

### P1-6 · Subscription reactivation state machine

**Report ref:** `MED-3`, persistent since v1.2.4.

> No `/suspend` path; cancelled subscriptions cannot be reactivated.

Add the `suspended` state and the transitions
`active → suspended → active` and `cancelled → suspended`.

### P1-7 · Tier deposit limits missing on enterprise

**Report ref:** `MED-DEPOSIT-LIMITS`, persistent since v1.2.4.

> 10M deposit accepted on enterprise — `deposit_limits["enterprise"]`
> still unset.

PR #85 added it for free/pro. Extend to enterprise per the policy
decision in the P0 plan.

### P1-8 · Slowloris regression check

**Report ref:** `MED-2`, possibly regressed in v1.2.7.

> Connection held 15.2s on sandbox, possibly Cloudflare-tolerated.

Validate on the sandbox with a local slow-body probe; Cloudflare may
be absorbing our mitigation so the test passed upstream but the origin
still holds the socket.

## P2 — nice to have

### P2-1 · Claims BOLA

**Report ref:** `LOW`, Security persona.

> `/v1/identity/agents/{any}/claims` returns 200 to any auth caller
> (empty body).

Empty body means no data leak, but the route should be 404 or 403 for
non-owners for consistency with the rest of the identity surface.

### P2-2 · RFC 6750 Bearer parsing compliance

**Report ref:** `LOW`, Security persona.

> `bearer`, double space, trailing space all parse on general
> endpoints (ownership checks still fire, so no priv-esc).

Tighten the parser to exactly `^Bearer [A-Za-z0-9._~+/=-]+$`. No
security impact but audit keeps flagging it.

### P2-3 · `/v1/metrics` RFC 9457 compliance

**Report ref:** `MED-10`, raised by SRE.

> Returns plain JSON 403, not RFC 9457.

Small consistency fix across the error surface.

### P2-4 · 2-decimal regex breaks crypto round-trip

**Report ref:** `MED-CRYPTO-REGEX`, raised by Web3.

> Regex enforces 2 decimals on amounts, breaks ETH/BTC precision.

Switch to the precision declared by the currency (2 for fiat, 8+ for
crypto).

### P2-5 · SDK paper cuts (7 v1.2.3-era items)

**Report ref:** Indie DX persona.

> Import name, float money, no currency arg, exception classes, async
> type hints — all persisting since v1.2.3.

Already tracked in `tasks/backlog/v1.2.4-audit-p1.md`; marking P2 here
for visibility but no new work.

### P2-6 · `/v1/me` or `/v1/auth/whoami` endpoint

**Report ref:** Indie DX persona.

> No way to introspect "who am I" with a key.

Nice DX improvement; trivial to implement.

## Wins in v1.2.7 (do not regress)

From the audit report, these must stay green:

- All 6 persistent v1.2.3/v1.2.4 CRITs on `/v1/infra/keys` fixed
  (admin gate, X-Forwarded-For spoof, X-Real-IP/Forwarded spoof,
  X-Api-Key ownership, bearer header normalization, path-normalization
  trailing slash).
- SDK 1.2.7 ISO8601 vs float crash fixed (`BeforeValidator(_parse_timestamp)`).
- `/v1/verification/submit` SDK workaround (routes to `/v1/gatekeeper/jobs`).
- Idempotency body-mismatch returns 409 (P0-4 from v1.2.4 PR).
- USD→ETH conversion restored (rate=0.0002500).
- Refund fee retention + decimal precision + disputes/webhook 201s.
- Backups path leak + onboarding quickstart.
- `/v1/execute` returns 410 (not 422) on legacy calls.

Add regression tests at the **sandbox-parity** layer for each of the
above so that v1.2.8 cannot silently roll them back.

## Suggested PR strategy

**One branch per P0 item**, merged via a shared integration branch
`fix/v1.2.7-audit-p0`. Reasoning: the budget-enforcement root cause
(P0-1) is blocking *every* paid endpoint test, so it MUST land first
and be green on sandbox-parity before the remaining P0s can even be
verified against the sandbox. Sequence:

1. P0-1 budget enforcement (unblocks all paid-endpoint tests)
2. P0-2 Gatekeeper Z3 OR pull from pricing
3. P0-3 `/v1/identity/agents` POST tier gate
4. P0-4 SDK field naming aliases
5. P0-5 SSE heartbeats
6. P0-6 restore/alias missing admin routes

Then open **one integration PR** with squash-merged P0s plus an
`audit-response-v1.2.7.md` narrative doc.
